# UIFactory

[![CI](https://github.com/chanthaphan/uifactory/actions/workflows/ci.yml/badge.svg)](https://github.com/chanthaphan/uifactory/actions/workflows/ci.yml)

UIFactory is a low-code **internal-app platform**. Builders sign in with SSO, add **connectors** (data
sources and agent APIs), and build multi-page apps — dashboards built by **drag-and-drop**, generated by
**AI**, or hand-edited as **HTML**, plus conversational **chat** pages backed by an LLM or an agent
connector. Apps are **versioned**, **deployed**, and **shared** with the organization. Admins manage
users, templates, prebuilt connectors, platform branding, and guardrails.

```
SSO login ─▶ (admin grants Build access) ─▶ build app (UI + chat pages, scoped to connectors) ─▶ version & deploy ─▶ share (private/org/public) ─▶ run it
```

## Features

- **SSO login** — Azure AD / Entra ID (OIDC) with Microsoft Graph for the org directory. A built-in
  **dev mock login** is used automatically when Azure isn't configured, so it runs locally.
- **Users & onboarding** — three platform roles: `admin` (manage users, templates, connectors, settings),
  `member` (a **builder** — create & edit apps), and `viewer`. New SSO users are provisioned as **viewer**
  (catalog access only); an admin promotes them to **member** to grant the Build tools. The first user /
  `ADMIN_EMAILS` become admin. Apps also have **per-app roles** (owner / editor / viewer).
- **Three ways to build a UI page**, switch freely between them:
  - **Drag-and-drop builder** — drop headings, text, KPI metrics, tables, charts, inputs, file uploads,
    buttons, and column containers onto a canvas and bind them to your data.
  - **AI generation** — describe the page (or click "add a component" suggestions); refine iteratively.
  - **Source code** — edit the page HTML directly with a live preview.
- **Chat pages** — a conversational assistant, optionally grounded on a query's data, with **markdown
  rendering**, **streaming** responses, and **per-user conversation history** (named, resumable threads).
- **Interactive UIs & write-back** — pages get a `window.UIFactory` bridge to run parameterized queries,
  refresh, navigate, **read user-selected files** (`UIFactory.readFile`), show alerts, copy/download, and
  submit forms that **write back** (INSERT/UPDATE/DELETE / POST).
- **Undo/redo** — the editor keeps an in-session history of edits to the app definition (pages, content,
  bindings, connector scope, theme), with **Undo/Redo** toolbar buttons.
- **Versioned publish & rollback** — editing happens on a draft; **Deploy/Publish** snapshots an
  immutable version. A **Versions** panel lists history and **restores any previous version** to the draft.
- **AI connection** — point generation and chat at the **platform LLM** or **this app's own provider key**
  (Claude / OpenAI / Azure OpenAI). External assistants are connected as **Agent API connectors** (below)
  and selected per chat page. **Build guidelines** (an AGENTS.md / CLAUDE.md-style note per app) steer
  both generation and chat.
- **Connectors** — add REST, PostgreSQL, SQLite, Microsoft 365, or **Agent API** connectors per app, from
  the editor's **Connectors** panel. Add one manually (with server-side, per-type validation; REST has a
  real "Test connection" probe) or **clone a prebuilt connector** an admin curated. **View/edit** any
  connector or query in place — secrets you leave masked are preserved on save. REST queries carry a
  **method, body, and schema/usage guidance** that steers the AI.
- **Per-page connector scoping** — each page can be restricted to a subset of the app's connectors; a
  hard runtime boundary blocks pages from running queries on out-of-scope connectors.
- **Per-user credentials & identity** — a data source can require **each user's own credentials**
  (encrypted per user), or **forward a signed user-identity assertion** (JWT) plus server-trusted
  `{{user_email}}` / `{{user_id}}` template values to downstreams.
- **Deploy, catalog & sharing** — deploy finished apps; permitted users run them from a catalog.
  Visibility is `private` (specific people), `org` (anyone signed in), or `public`.
- **Admin console** — manage users & roles, templates (incl. from an existing app), prebuilt connectors,
  platform **name / logo / brand color**, and AI defaults. The browser **tab title + favicon** follow the
  configured platform name/logo.
- **Guardrails** — per-page connector scoping + query allowlist, builder-only authoring, app-size caps,
  per-user rate limits, chat input/length limits, and conversation retention (all env-tunable — see
  [docs/configuration.md](docs/configuration.md)).

## Prerequisites

- Node.js **22.5+** (uses the built-in `node:sqlite` module)

## Setup

```bash
# from the repo root
npm install            # installs backend + frontend workspaces
npm run setup          # prisma generate + db push + seed
```

The seed creates a demo SQLite database (`backend/data/sample.db`) with customers/products/orders **and
accounts/transactions**, plus **10 bank- and agent-focused templates**, **3 prebuilt connectors**, a
deployed sample app ("Retail Banking Console"), and **three demo users**:

| Email | Role |
| ----- | ---- |
| `admin@uifactory.local` | admin |
| `alice@uifactory.local` | member |
| `bob@uifactory.local`   | member |

## Run

```bash
npm run dev            # backend (:3001) + frontend (:5173)
```

Open **http://localhost:5173**. With no Azure config you'll see the **dev login** — pick a demo user.

## Test

```bash
npm test --workspace backend   # runs the backend unit tests (node --test via tsx)
```

Unit tests cover the security-critical pure functions (per-page connector scope, secret redaction/merge,
mutation detection, SSRF IP classification, config schemas, template remap).

## Upgrading

When upgrading an existing deployment, run the one-off migration that converts any app-level
"external agent API" connections into Agent API connectors (idempotent):

```bash
npm run migrate:agent-connectors --workspace backend
```

## Configure SSO (Azure AD / Entra ID)

Set all four in `backend/.env` to switch from dev mock to real Microsoft sign-in (and Graph directory):

```env
AZURE_AD_TENANT_ID=...
AZURE_AD_CLIENT_ID=...
AZURE_AD_CLIENT_SECRET=...
AZURE_AD_REDIRECT_URI=http://localhost:3001/api/auth/callback
# emails granted admin on first login (first-ever user also becomes admin)
ADMIN_EMAILS=you@yourtenant.com
JWT_SECRET=change-me
```

## Configure AI

The platform LLM is set via env (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, or the `AZURE_OPENAI_*` vars;
auto-detected, or forced with `AI_PROVIDER`). Individual apps override the LLM in **App → Settings → AI
connection** with their own provider key. To use an external assistant, add an **Agent API connector** and
select it on a chat page. Without any provider, chat and generation fall back to a clearly-labelled mock /
built-in template. See [docs/configuration.md](docs/configuration.md) for all variables.

## Security highlights

- **Tenant isolation** — data sources, queries, and credentials are scoped to their app and caller.
- **Secrets at rest** — data source configs, per-app AI/agent keys, and per-user credentials are
  AES-256-GCM encrypted (`SECRETS_KEY`) and redacted in API responses.
- **Builder-only authoring** — creating/editing/deploying apps requires the `admin` or `member` role;
  viewers can only run/view shared apps. Role denials return `403`.
- **Per-page query allowlist + connector scope** — at runtime a page may only run its own bound query +
  actions, and (when scoped) only on its allowed connectors; parameters are bound (no string interpolation
  into SQL). Connector configs are validated server-side per type on create/update.
- **Identity, not spoofable** — `{{user_*}}` values and the signed `X-UIFactory-User` assertion are
  injected from the authenticated session; the sandboxed UI cannot set them.
- **SSRF protection** — REST/agent/LLM URLs that resolve to private/reserved IPs are blocked
  (`ALLOW_PRIVATE_NETWORK` / `OUTBOUND_ALLOWLIST` to override).
- **Sandboxed rendering** — generated UI runs in a sandboxed iframe (opaque origin) and reaches the host
  only through the `window.UIFactory` postMessage bridge — it can't call the API or read cookies directly.
