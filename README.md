# UIFactory

[![CI](https://github.com/chanthaphan/uifactory/actions/workflows/ci.yml/badge.svg)](https://github.com/chanthaphan/uifactory/actions/workflows/ci.yml)

UIFactory is a low-code **internal-app platform**. Members sign in with SSO, connect data sources and
agent APIs, and build multi-page apps — dashboards built by **drag-and-drop**, generated by **AI**, or
hand-edited as **HTML**, plus conversational **chat** pages backed by an LLM or your own agent. Apps
are **versioned**, **deployed**, and **shared** with the organization. Admins manage users, templates,
prebuilt connectors, platform branding, and guardrails.

```
SSO login ─▶ build app (UI + chat pages, bound to data/agents) ─▶ version & deploy ─▶ share (private/org/public) ─▶ run it
```

## Features

- **SSO login** — Azure AD / Entra ID (OIDC) with Microsoft Graph for the org directory. A built-in
  **dev mock login** is used automatically when Azure isn't configured, so it runs locally.
- **Users & roles** — `admin` (manage users, templates, connectors, settings) and `member` (build &
  use apps), plus **per-app roles** (owner / editor / viewer). Users are auto-provisioned on first login.
- **Three ways to build a UI page**, switch freely between them:
  - **Drag-and-drop builder** — drop headings, text, KPI metrics, tables, charts, inputs, file uploads,
    buttons, and column containers onto a canvas and bind them to your data.
  - **AI generation** — describe the page (or click "add a component" suggestions); refine iteratively.
  - **Source code** — edit the page HTML directly with a live preview.
- **Chat pages** — a conversational assistant, optionally grounded on a query's data, with **markdown
  rendering**, **streaming** responses, and **per-user conversation history** (named, resumable threads).
- **Interactive UIs & write-back** — pages get a `window.UIFactory` bridge to run parameterized queries,
  refresh, navigate, **read user-selected files** (`UIFactory.readFile`), show alerts, copy/download, and
  submit forms that **write back** (INSERT/UPDATE/DELETE / POST).
- **Versioned publish & rollback** — editing happens on a draft; **Deploy/Publish** snapshots an
  immutable version. A **Versions** panel lists history and **restores any previous version** to the draft.
- **AI & agent connections** — point generation and chat at the **platform LLM**, **this app's own
  provider key** (Claude / OpenAI / Azure OpenAI), or an **external agent API** (your own assistant /
  agent-builder endpoint, which may return dynamic text/JSON the UI decodes). **Build guidelines**
  (an AGENTS.md / CLAUDE.md-style note per app) steer both generation and chat.
- **Data sources & prebuilt connectors** — REST, PostgreSQL, SQLite, or Microsoft 365 per app. REST
  queries carry a **method, body, and schema/usage guidance** that steers the AI. Admins curate
  **prebuilt connectors** any member can clone in one click.
- **Per-user credentials & identity** — a data source can require **each user's own credentials**
  (encrypted per user), or **forward a signed user-identity assertion** (JWT) plus server-trusted
  `{{user_email}}` / `{{user_id}}` template values to downstreams.
- **Deploy, catalog & sharing** — deploy finished apps; permitted users run them from a catalog.
  Visibility is `private` (specific people), `org` (anyone signed in), or `public`.
- **Admin console** — manage users, templates (incl. from an existing app), prebuilt connectors,
  platform **name / logo / brand color**, and AI defaults.
- **Guardrails** — per-page data isolation, app-size caps, per-user rate limits, chat input/length
  limits, and conversation retention (all env-tunable — see [docs/configuration.md](docs/configuration.md)).

## Prerequisites

- Node.js **22.5+** (uses the built-in `node:sqlite` module)

## Setup

```bash
# from the repo root
npm install            # installs backend + frontend workspaces
npm run setup          # prisma generate + db push + seed
```

The seed creates a demo SQLite database (`backend/data/sample.db`) with customers/products/orders **and
accounts/transactions**, plus **10 bank- and agent-focused templates**, **3 prebuilt connectors**, a
deployed sample app ("Retail Banking Console"), and **three demo users**:

| Email | Role |
| ----- | ---- |
| `admin@uifactory.local` | admin |
| `alice@uifactory.local` | member |
| `bob@uifactory.local`   | member |

## Run

```bash
npm run dev            # backend (:3001) + frontend (:5173)
```

Open **http://localhost:5173**. With no Azure config you'll see the **dev login** — pick a demo user.

## Configure SSO (Azure AD / Entra ID)

Set all four in `backend/.env` to switch from dev mock to real Microsoft sign-in (and Graph directory):

```env
AZURE_AD_TENANT_ID=...
AZURE_AD_CLIENT_ID=...
AZURE_AD_CLIENT_SECRET=...
AZURE_AD_REDIRECT_URI=http://localhost:3001/api/auth/callback
# emails granted admin on first login (first-ever user also becomes admin)
ADMIN_EMAILS=you@yourtenant.com
JWT_SECRET=change-me
```

## Configure AI

The platform LLM is set via env (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`, or the `AZURE_OPENAI_*` vars;
auto-detected, or forced with `AI_PROVIDER`). Individual apps override this in **App → Settings → AI /
agent connection** with their own provider key or an external agent API. Without any provider, chat and
generation fall back to a clearly-labelled mock / built-in template. See
[docs/configuration.md](docs/configuration.md) for all variables.

## Security highlights

- **Tenant isolation** — data sources, queries, and credentials are scoped to their app and caller.
- **Secrets at rest** — data source configs, per-app AI/agent keys, and per-user credentials are
  AES-256-GCM encrypted (`SECRETS_KEY`) and redacted in API responses.
- **Per-page query allowlist** — at runtime a page may only run its own bound query + its own actions;
  parameters are bound (no string interpolation into SQL).
- **Identity, not spoofable** — `{{user_*}}` values and the signed `X-UIFactory-User` assertion are
  injected from the authenticated session; the sandboxed UI cannot set them.
- **SSRF protection** — REST/agent/LLM URLs that resolve to private/reserved IPs are blocked
  (`ALLOW_PRIVATE_NETWORK` / `OUTBOUND_ALLOWLIST` to override).
- **Sandboxed rendering** — generated UI runs in a sandboxed iframe (opaque origin) and reaches the host
  only through the `window.UIFactory` postMessage bridge — it can't call the API or read cookies directly.
